# ICP, Wedge, KPI, And Sales Narrative

This document defines the sharpest initial market entry for Tenio.

It is meant to keep the company focused while the product matures.

## Core Decision

Do not broaden Tenio into generic healthcare admin automation or a horizontal AI workflow platform.

The right move is to stay in healthcare and narrow further.

The category should remain:

- workflow OS for claim-status work
- powered by automation

## Best Initial ICP

The best initial customer profile is:

- mid-sized outsourced RCM firms
- AR follow-up teams at multi-site provider groups
- operations leaders with visible backlog and SLA pain
- teams spending real labor hours in payer portals, spreadsheets, and ad hoc handoffs

These customers are attractive because they already:

- feel the pain every day
- have budget tied to operational performance
- have teams dedicated to follow-up work
- can measure ROI in labor, backlog, and cycle-time terms

## Best Initial Buyer

Start with one buyer, not many.

The strongest first buyers are:

- outsourced RCM or BPO leaders
- heads of revenue cycle
- AR managers
- RCM operations leaders

They are not buying AI for its own sake.

They are buying:

- control
- visibility
- throughput
- auditability
- operational reliability

## Best Initial User

The day-to-day users are:

- claim status representatives
- AR follow-up specialists
- denial and escalation specialists
- team leads

Managers are the second user class, because they need to govern:

- backlog
- SLA risk
- review load
- bottlenecks
- team throughput

## Best Initial Wedge

The wedge should be narrow and painful:

- claim-status follow-up
- with evidence capture
- routing
- ownership
- SLA visibility

In plain language:

Tenio should be the place claim-status work gets done.

That means the first wedge is not:

- all of RCM
- all of healthcare admin
- all payer communications
- every denial workflow

It is:

- one workflow
- one buyer
- one labor pool
- one measurable pain point

## The Exact Job To Win

Tenio should win this job first:

"Run claim-status follow-up in one governed workflow instead of across payer portals, spreadsheets, notes, and disconnected queues."

That is a better wedge than:

- generic "healthcare admin automation"
- generic "AI agents for providers"
- broad "revenue cycle OS"

## Why This Wedge Works

It works because it combines:

- obvious manual pain
- repetitive work suitable for automation
- a real workflow coordination problem
- clear manager visibility needs
- enterprise trust requirements around evidence and auditability

This is exactly where Tenio's workflow-plus-automation structure is strongest.

## The KPI To Sell First

Pick one primary KPI and two supporting ones.

Primary KPI:

- fewer touches per claim

Supporting KPIs:

- faster follow-up cycle time
- better SLA compliance

Secondary proof metrics:

- lower review time per claim
- better backlog visibility
- fewer escalations caused by unclear ownership
- more claims worked per rep

Do not try to sell too many outcomes at once.

The easiest commercial story is:

- Tenio reduces manual touches
- while improving control and visibility

## The First Sales Narrative

Use this structure in conversations:

### 1. Start with the pain

Claim-status work is fragmented across payer portals, spreadsheets, PM systems, notes, and human memory.

That creates:

- repeated manual work
- poor ownership clarity
- inconsistent interpretation
- missed SLAs
- weak auditability
- poor manager visibility

### 2. Name the category clearly

Tenio is the workflow OS for claim-status work.

It is where queue, ownership, routing, review, evidence, and auditability live.

### 3. Explain the automation layer correctly

Tenio automates payer retrieval and first-pass interpretation so reps focus on exceptions.

Automation is not the system of record.

The workflow layer is.

### 4. Land the ROI

Customers get:

- fewer manual portal touches
- faster exception routing
- cleaner ownership
- better backlog visibility
- stronger SLA control
- evidence attached to the work itself

### 5. Make the rollout feel safe

Customers do not need a giant migration or platform replacement.

They can:

1. import active inventory
2. start with one workflow
3. prove value
4. expand to more queues and payers

## The Tight One-Liner

Tenio is the workflow OS for claim-status work, powered by automation that retrieves payer status, attaches evidence, and routes uncertain cases to human review.

## The Positioning Guardrails

Do not say:

- we automate healthcare admin
- we are a horizontal agent platform
- we are a broad revenue OS

Do say:

- we run claim-status follow-up work
- we give RCM teams one governed workflow
- we automate the portal work underneath

## The Expansion Rule

Only expand after Tenio has:

- 3 to 5 paying customers
- one repeatable buyer
- one repeatable workflow wedge
- one hard proof metric

The expansion path should be:

1. more claim-status volume
2. more payer coverage
3. more adjacent RCM workflows
4. only later, broader healthcare workflows

## Practical Recommendation

If forced to choose the sharpest initial GTM lane, choose:

- buyer: outsourced RCM / BPO leader
- user: claim-status follow-up reps and managers
- wedge: claim-status workflow with evidence, routing, and SLA control
- KPI: fewer touches per claim

That is the most believable, measurable, and strategically coherent starting point for Tenio.
