# Design Partner Launch

This document defines the first-customer launch motion for Tenio.

## Goal

Validate that Tenio can run a bounded but real claim-status workflow in production-like conditions with a design partner.

## Ideal First Customer

Prioritize teams with:

- high manual payer portal usage
- unresolved claim backlog
- measurable SLA pressure
- fragmented ownership and review processes
- willingness to centralize work in a new workflow layer

Good early targets:

- revenue cycle teams with large follow-up queues
- AR managers with operational visibility problems
- BPO teams handling repetitive claim-status workflows

## Launch Scope

Keep the first launch narrow.

The initial scope should define:

- one claim-status workflow
- one active inventory segment
- one connector path or retrieval path
- one manager sponsor
- one operator cohort

## Launch Sequence

### 1. Discovery

Document:

- current tools
- current handoffs
- ownership rules
- review rules
- SLA expectations
- the customer's current pain metrics

### 2. Data And Workflow Setup

- map customer data into Tenio
- import a bounded active inventory
- configure the initial workflow and ownership rules
- confirm the first reporting and audit expectations

### 3. Pilot Go-Live

- start working the selected workflow in Tenio
- keep launch support tight and visible
- monitor queue health, retrieval issues, and operator adoption daily

### 4. Success Review

Review:

- throughput
- review load
- backlog visibility
- SLA performance
- retrieval reliability
- operator and manager trust

### 5. Expansion

After the bounded workflow is healthy:

- add more claims or queues
- add more connector coverage
- add more managers and operators
- expand downstream integrations

## Success Criteria

Treat a design-partner launch as successful when:

- operators can work real claims in Tenio every day
- managers trust the queue, evidence, and audit history
- retrieval automation removes meaningful manual work
- the customer wants to expand the workflow footprint

## Metrics To Track

- claims worked in Tenio
- retrieval success rate
- claims routed to review
- average time to resolution
- backlog volume and aging
- SLA risk rate
- manager adoption of reporting views

## Feedback Loop

Capture feedback from:

- operators
- team leads
- managers
- implementation owners

Convert that feedback into:

- product backlog updates
- connector improvements
- onboarding playbook changes
- sharper GTM messaging

## Operating Rule

Do not try to prove everything in the first launch.

Prove that one workflow can run better in Tenio than it does in fragmented tools today.
